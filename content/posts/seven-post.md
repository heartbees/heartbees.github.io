# 一、为什么选Hugo？新手核心考量

纠结过WordPress、Hexo后，最终选Hugo，关键原因很直观：

- 速度快：本地构建仅需几十毫秒，预览修改实时刷新，无需久等；
- 易上手：环境依赖简单，安装后修改配置文件即可使用；
- 零成本托管：可直接部署到GitHub Pages，无需付费服务器；
- Markdown友好：支持完整语法，专注内容创作无需纠结排版。

新手想快速搭建免费稳定的个人博客，Hugo值得尝试。

# 二、核心避坑：5个致命问题+解决方案

以下都是实操中遇到的问题，解决方案可直接抄作业！

## 坑1：部署脚本运行后“消失”
现象：脚本功能正常，但运行后文件夹中找不到，误以为被删除。
原因：脚本仅在main分支，运行时切换到gh-pages分支（无该脚本），Git隐藏了文件而非删除。
解决：用无分支切换的部署脚本；文件“消失”时直接搜索文件名即可找到；将脚本加入杀毒软件白名单防误删。

## 坑2：本地预览不显示新文章
现象：新文章已提交至GitHub，但运行hugo server后列表页无显示。
解决：① 把Front Matter中draft设为false（默认不渲染草稿）；② 日期改为当前/过去时间（未来时间会被跳过）；③ 删public/和resources/文件夹清缓存，重启hugo server -D；④ 文章必须放在content/posts/目录。

## 坑3：构建时报“语法错误”
现象：运行hugo命令报错，指向date字段，提示“could not find flow map content”。
原因：YAML格式不支持Go模板语法，如直接写date: {{ .Date }}，且格式不规范。
解决：date用标准格式（2026-01-15 20:00:00）；冒号后必须加空格；分类标签用2个空格缩进，禁用Tab。

## 坑4：线上不显示新文章，本地正常
现象：本地预览正常，部署到GitHub Pages后无新文章。
解决：① 重新运行部署脚本，确保gh-pages推送成功，失败则手动执行git subtree push --prefix=public origin gh-pages；② 检查仓库设置，Pages的Source需选gh-pages分支和根目录；③ 耐心等10-15分钟，用隐私模式刷新避缓存。

## 坑5：脚本中文乱码、提示无效命令
现象：中文提示显示乱码，CMD报错“不是内部或外部命令”。
原因：脚本保存为“UTF-8 带 BOM”格式，与CMD不兼容。
解决：VS Code中切换编码为“UTF-8 无 BOM”；保存时选Windows (CRLF)行尾序列。
